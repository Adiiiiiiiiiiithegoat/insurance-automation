# Motor Insurance Automation — Full Project Brief

## Company Background
Online insurance broker company based in Oman. Deals with motor insurance policies. Has 6 insurance provider companies. 3 of them have APIs already integrated. The remaining 3 require manual processing, which this automation aims to replace entirely.

---

## The Problem Being Solved
For the 3 non-API insurance companies, staff currently
1. Manually read Arabic documents (Mulkiya and License) uploaded by customers
2. Cross-reference document data against customer-submitted data
3. Log into each insurer's website manually
4. Fill in a detailed form field by field
5. Handle page reloads throughout the form
6. Approve the policy
7. Download the resulting PDF
8. Upload the PDF to the broker platform
9. Send the PDF to the customer via WhatsApp

This takes 10–15 minutes per policy. Volume is 30–70 policies per day across the 3 insurers. The goal is to reduce staff involvement to only flagged exceptions — cases where document data does not match submitted data.

---

## The Developer
Non-technical solo developer. No coding background. Claude Pro  Codex is being used to write all code. 

---

## Proof of Concept — Already Completed
A basic Playwright automation has already been successfully built and tested. It fills a form on a website using data from broker website. This proves browser automation works in this context. The next stage is building the full production system on top of this foundation.

## The Insurer Website — Already Analysed
One insurer website has been reviewed in detail. Key findings
- All field labels are in clear English
- Standard HTML form elements throughout — text boxes, dropdowns, date pickers
- No CAPTCHA present
- No OTP or two-factor login required
- Two Get Record auto-fill buttons that significantly reduce manual field entry
  - Get Licence Information — enter License No, click Get Record, website auto-populates customer fields
  - Get Vehicle Information — enter plate number, click Get Record, website auto-populates vehicle fields
- Some fields use a lookup popup triggered by a ≡ icon — these need specific handling
- Page reloads occur during form filling and must be handled with wait steps
- A Calculate button and a Validate button must be clicked at specific points
- Only specific fields need to be filled, not the entire form

## The Full Production Workflow

### Step 1 — Trigger
A staff member identifies a policy that needs processing on the broker portal. The automation is triggered for that specific record. This is the only manual touchpoint in the ideal flow.

### Step 2 — Navigate Broker Portal
The script logs into the broker portal and follows this exact navigation path
- Go to Payment by Channel page
- Filter for Payment Done cases
- Click the relevant subpart
- Set date range to one week before today and one week after today
- Click Search
- Click the eye icon on the specific record to open it

### Step 3 — Extract Data From Broker Portal
The script reads all relevant text fields from the opened record. These are the fields that will be used to fill the insurer form and to cross-reference against the documents.

### Step 4 — Open and Read Mulkiya (Arabic OCR)
The Mulkiya is attached to the record as an image or PDF. The script accesses this file and sends it to the Claude API. Claude extracts all relevant Arabic fields including
- Plate number
- Chassis number
- Civil ID
- Owner name
- Vehicle details
- Any other fields present on the document

Claude uses the claude-sonnet model for this step specifically due to the accuracy required for Arabic character recognition. A wrong chassis number or civil ID on an insurance policy is a serious business error.

### Step 5 — Open and Read License
Same process as the Mulkiya. The License is sent to Claude API for extraction of relevant fields including driver details and license information.

### Step 6 — Cross-Verification
Claude compares the extracted document values against the customer-submitted data from the broker portal. If all values match, the process continues automatically. If any value mismatches, the automation stops, logs the specific mismatch clearly, and sends an alert to the staff member to resolve it manually. No incorrect data ever reaches the insurer.

### Step 7 — Open Insurer Website
The script opens a new browser tab or window and navigates to the insurer website. It logs in using stored credentials.

### Step 8 — Fill the Insurance Form
The script fills the form intelligently
- Uses Get Licence Information button to auto-populate customer fields — enters License No and clicks Get Record
- Uses Get Vehicle Information button to auto-populate vehicle fields — enters plate number and clicks Get Record
- Fills remaining required fields manually using verified data
- Some fields are fixed values that never change across all policies (e.g. broker phone number, location name) — these are hardcoded in the script
- Handles page reloads by waiting for the next section to load before continuing
- Handles lookup popups triggered by ≡ icons
- Clicks Calculate when required
- Clicks Validate when required

### Step 9 — Approve and Save PDF
Once the form is complete and premiums are calculated, the script approves the policy and downloads the generated PDF. The PDF is saved with a filename tied to the policy number.

### Step 10 — Return to Broker Portal
The script navigates back to the broker portal record and uploads the PDF along with any additional required details to confirm the policy.

### Step 11 — WhatsApp Delivery
n8n detects the new confirmed policy and automatically sends the PDF to the customer's WhatsApp number via the WhatsApp Business Cloud API.

## Technical Stack

### Core Automation
Browser Use — AI-powered browser automation library built on top of Playwright. Unlike traditional automation that follows fixed recorded steps, Browser Use uses an AI model to read each page and decide the next action contextually. This makes it resilient to minor website changes and handles dynamic page behaviour like reloads far better than recording-based tools.

Python — the programming language that ties everything together. Runs on the cloud server. Calls the Claude API, controls Browser Use, handles files, triggers n8n. All Python code is written by Claude Pro — the developer does not write any code manually.

Playwright — the underlying browser control library that Browser Use is built on. Already proven working in the proof of concept phase.

### AI  OCR
Claude API — used inside the automation pipeline for document reading and cross-verification. Two models are used
- claude-sonnet for Mulkiya and License OCR — higher accuracy for Arabic, used once per policy
- claude-haiku for Browser Use decision making — cheaper, fast enough for form navigation, used many times per policy

The Claude API is separate from Claude Pro. It is called programmatically by the Python script with no human involvement.

### Delivery
WhatsApp Business Cloud API — Meta's official API for sending WhatsApp messages programmatically. Connected through n8n. First 1000 conversations per month are free. Beyond that approximately $0.05–0.08 per conversation.

### Infrastructure
Cloud server — approximately $5–7 per month. Runs Linux. Hosts Python scripts, Browser Use, and n8n. Runs 24 hours a day. All code lives here. The developer connects to it via terminal when maintenance is needed, with Claude guiding every command.

## Data Sources Per Field

Fields on the insurer form come from one of four sources
1. Broker portal — scraped directly from the open record page
2. Mulkiya — extracted by Claude OCR from the Arabic document
3. License — extracted by Claude OCR
4. Fixed values — hardcoded in the script, same for every policy (e.g. broker phone number, location name, certain dropdown defaults)

The exact field-to-source mapping for the specific insurer has not yet been fully documented and needs to be completed before the browser automation script can be written.
